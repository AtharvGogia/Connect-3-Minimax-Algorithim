SLIDE 2: INTRODUCTION & FORMAL DEFINITION
- Content:
    * Concept: A web-based pedagogical tool for visualizing state-space search.
    * Formal Definition of a Game:
        - Initial State (S0): The empty N x N grid.
        - PLAYER(s): MAX (Computer) and MIN (Human).
        - ACTIONS(s): Set of legal moves in state s.
        - RESULT(s, a): Transition model (state after move a).
        - TERMINAL-TEST(s): Boolean check for game end.
        - UTILITY(s, p): Payoff function (+1 for MAX win, -1 for MIN win, 0 for draw).
- Visuals/Screenshots:
    * [SCREENSHOT 2]: A collage of the three grid sizes (3x3, 4x4, 5x5).
    * [DIAGRAM]: A block diagram showing the formal game elements flow.
- Speaker Notes: "Connect X is built on the formal definition of a game. We model it as a zero-sum environment where one player's gain is the other's loss. We track every state from S0 to the terminal state using a transition model."

SLIDE 3: THE CHALLENGE: STATE-SPACE COMPLEXITY
- Content:
    * The Problem: Exponential growth of possible moves.
    * Branching Factor (b): Number of legal moves at any given turn.
    * Depth (m): Maximum depth of the tree.
    * Time Complexity: O(b^m).
    * Space Complexity: O(bm) using Depth-First Exploration.
    * Requirement: Efficient search and pruning are mandatory for real-time performance on 5x5 grids.
- Visuals/Screenshots:
    * [SCREENSHOT 3]: A screenshot of a dense search tree zoomed out.
    * [TEXT OVERLAY]: "O(b^m) - The Curse of Dimensionality".
- Speaker Notes: "In AI, we face the 'Curse of Dimensionality'. As the board grows, the number of paths to victory explodes. Without optimization, a computer would take minutes to make a single move. We solve this using Minimax and Alpha-Beta Pruning."

SLIDE 4: ALGORITHM I: THE MINIMAX STRATEGY
- Content:
    * Logic: Recursive "Look-Ahead" strategy for deterministic, 2-player games.
    * Goal: Identify the best achievable payoff against best play.
    * Maximizing Player (MAX): Tries to maximize its score.
    * Minimizing Player (MIN): Tries to minimize MAX's score.
    * Properties:
        - Complete? Yes (if tree is finite).
        - Optimal? Yes (against an optimal opponent).
- Visuals/Screenshots:
    * [SCREENSHOT 4]: A close-up of the tree visualization showing a "Leaf Node" score being passed to its parent.
    * [ANNOTATION]: Use "MAX" and "MIN" labels on alternating levels.
- Speaker Notes: "Minimax assumes 'Rationality'. The computer looks 5 moves ahead and picks the 'best of the worst' outcomes. It assumes you will play perfectly to stop it, so it identifies the highest minimax value it can guarantee."

SLIDE 5: ALGORITHM II: ALPHA-BETA PRUNING
- Content:
    * Optimization: Eliminates branches that cannot influence the final decision.
    * Thresholds:
        - Alpha (α): Best (highest) value found so far for MAX.
        - Beta (β): Best (lowest) value found so far for MIN.
    * Pruning Condition: If α ≥ β, the branch is discarded.
    * Efficiency: Reduces evaluated nodes significantly without losing optimality.
- Visuals/Screenshots:
    * [SCREENSHOT 5]: A clear shot of a "Pruned" branch (dashed line with an 'X').
    * [LOG CAPTURE]: A snippet from the "System Logs" showing: "Pruning branch: Alpha (10) >= Beta (10)".
- Speaker Notes: "Alpha-Beta Pruning is our secret weapon. If the computer finds a move in one branch that is already better than anything it could find in another, it simply stops looking. It 'prunes' the tree, saving massive amounts of memory and time."

SLIDE 6: MOVE ORDERING & PERFORMANCE
- Content:
    * Effectiveness: Highly dependent on the order in which nodes are examined.
    * Worst Ordering: Best move occurs on the right; complexity remains O(b^m).
    * Ideal Ordering: Best move occurs on the left; complexity reduced to O(b^{m/2}).
    * Result: Ideal ordering effectively doubles the searchable depth in the same amount of time.
- Visuals/Screenshots:
    * [SCREENSHOT 6]: A high-action shot mid-calculation with multiple glowing nodes.
    * [GRAPH]: A simple comparison of nodes searched with vs. without pruning.
- Speaker Notes: "Pruning efficiency depends on 'Move Ordering'. By evaluating promising moves first, we aim for the 'Ideal Ordering' complexity of O(b to the power of m/2). This is why we can handle a 5x5 grid in real-time while others might freeze."

SLIDE 7: TECHNICAL INNOVATION: GENERATORS & ASYNC UI
- Content:
    * The Problem: Heavy recursion freezes the browser's main thread.
    * The Solution: JavaScript Generators (function*).
    * Mechanism: The minimax function yields after every node evaluation.
    * Result: A smooth, cinematic visualization of a process that usually happens invisibly in milliseconds.
- Visuals/Screenshots:
    * [SCREENSHOT 7]: A screenshot of the minimaxGenerator code highlighting the yield keyword.
    * [DIAGRAM]: A "Generator -> UI Update -> Resume" loop diagram.
- Speaker Notes: "One of our biggest technical achievements was making recursion 'visible'. Usually, recursion is a 'black box'. By using Generators, we effectively 'slowed down' time, allowing the browser to draw the tree frame-by-frame without crashing."

SLIDE 8: HEURISTIC EVALUATION (STATIC EVALUATOR)
- Content:
    * Concept: Used when it's not feasible to traverse the entire tree.
    * Purpose: Provides an estimate of utility at a given non-terminal position.
    * Scoring Rules:
        - +1000 for Win, -1000 for Loss.
        - Points for "Open 3-in-a-row" and "Center Control".
    * Advantage: Allows the agent to make intelligent moves even when the goal is not yet reachable.
- Visuals/Screenshots:
    * [SCREENSHOT 8]: A screenshot of a 5x5 game in progress showing tokens clustered in the center.
    * [CODE SNIPPET]: The evaluate function scoring logic.
- Speaker Notes: "Since we can't search to the very end of every game, we use a 'Heuristic Evaluation Function'. It's like an intuition—it scores the board based on potential. A line of 3 tokens is worth more than 2, guiding the AI toward a win before it's actually achieved."

SLIDE 9: PERFORMANCE ANALYSIS & CONCLUSION
- Content:
